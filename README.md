# 报价更新项目迁移说明（给其他 Agent 软件）

## 1. 项目用途
- 用于更新项目报价文件中的：
  - 网价：`G1/G3/G4`
  - 图片/文档价：`H1/H3/H4`
- 支持：
  - 单文件更新
  - 批量更新
  - `web` / `image_doc` / `both` 三种模式

### 1.1 线下图片识别主链路说明（2026-05-11）
- 图片/PDF 默认只走 `MiniMax VLM` 结构化识别主链路。
- 传统 OCR 不再参与默认执行流程，仅保留历史诊断用途。
- 模型输出会进入确定性规则校验（价格范围、目标地点、库存状态），校验失败会阻断回写并进入人工确认。

## 2. 目录结构（核心）
- `ocr_price/`：核心业务代码
- `skills/quote-update/`：执行脚本、流程规则、命令参考
- `requirements-ocr.txt`：Python 依赖
- `.env.example`：环境变量模板
- `网站账号密码.txt`：网价站点账号密码（或在新环境创建同名文件）
- `项目报价/`：待更新项目 Excel
- `线下报价/`：图片/文档原始来源
- `运行产物/`：报告、对照表、中间产物

## 3. 新环境准备
在项目根目录执行：

```powershell
pip install -r requirements-ocr.txt
playwright install chromium
```

然后：
1. 复制 `.env.example` 为 `.env` 并填写配置。
2. 确认 `网站账号密码.txt` 可用（或用命令行参数传用户名密码）。

## 4. 常用命令

### 单文件（默认全流程）
```powershell
python skills/quote-update/scripts/run_single.py `
  --project "项目报价/<项目Excel路径>" `
  --mode both `
  --headless
```

### 批量（默认全流程）
```powershell
python skills/quote-update/scripts/run_batch.py `
  --project-dir "项目报价" `
  --glob "*.xlsx" `
  --mode both `
  --headless
```

### 只更新网价
把 `--mode both` 改为 `--mode web`。

### 只更新线下报价
把 `--mode both` 改为 `--mode image_doc`。

### 安全执行（先预演，再写入）
默认必须先运行 dry-run：

```powershell
python skills/quote-update/scripts/run_single.py `
  --project "项目报价/<项目Excel路径>" `
  --mode both `
  --dry-run `
  --headless
```

确认报告无待确认/无异常后，再允许真实写入：

```powershell
python skills/quote-update/scripts/run_single.py `
  --project "项目报价/<项目Excel路径>" `
  --mode both `
  --confirm-write `
  --headless
```

`--confirm-write` 默认直接复用最近一次 dry-run 产物做 apply 写入（不重复抓网价、不重复跑OCR）。如确需重生产物，可追加：
- `--refresh-web-artifacts`
- `--refresh-image-artifacts`

线下报价价格校验以项目文件中同厂家 sheet 的网价为动态参考：H3(盘螺) 对比 G3，H4(螺纹) 对比 G4。任一价格与网价的差值超过 1000 元/吨或偏离比例超过 20% 时，不自动回写，报告为“线下价与网价偏差过大”，需要人工确认。若 G3/G4 没有可用网价，允许回写，但报告备注“无网价参考”。

### 低能力模型/外部Agent执行限制
低能力模型不得直接修改 Excel，也不得手动拼写单元格更新。统一入口是 `skills/quote-update/scripts/run_single.py` 或 `run_batch.py`。默认先 `--dry-run`，用户确认后才可 `--confirm-write`。任何待确认、异常偏差、缺少网价参考、电议或脚本错误都必须停止并报告。

#### 5.0.2 格式要求详解

**1. 标题规范**
- 单文件更新：使用"单文件"；批量更新：使用"批量"
- 必须包含执行时间和更新模式

**2. 价格对比格式**
- 格式：`老值 → 新值`
- 示例：`3690 → 3690` 或 `空 → 3620` 或 `3490 → 空`
- 即使价格未变化，也必须显示对比格式
- 日期格式：`网价[YYYY-MM-DD]` 或 `报价[YYYY-MM-DD]`

**3. 表格列说明**
- **G1/H1**：日期列，格式固定
- **G3/H3**：盘螺价格（Φ8-10）
- **G4/H4**：螺纹价格（Φ18）

**4. 未更新原因格式**
- 格式：`跳过（X个）：[厂家名1、厂家名2] - [原因]`
- 原因分类：
  - `未匹配(不更新)`：厂家未在对照表中确认匹配
  - `未提取到有效价格`：网价或文档中无有效价格
  - `跳过(汇总页)`：报价表的汇总Sheet
  - `登录失败`：网价站点登录失败
  - `网络异常`：网络问题导致抓取失败

**5. 汇总统计表**
- 必须按项目分列统计
- 最后一行显示合计
- 统计已更新和未更新的厂家数量

**6. 无新增厂家告知**
- 本次执行若未发现新厂家，必须在开头添加：
  > ### 📌 复用确认记录说明
  > 本次执行未发现新厂家，所有对照关系已复用历史确认记录，无需额外操作。

### 5.1 完整执行流程（必须按序执行）

#### 5.1.1 标准执行步骤

```
0. 解析用户意图
   - 仅说"更新价格/更新报价"时 → 默认 both（网价 + 图片/文档）
   - 明确限制范围时 → 切换为 web 或 image_doc 单链路

1. 识别项目文件地点（从文件名提取）

2. 网价流程（web/both）
   2.1 登录网价站点
   2.2 查询最新价格清单（按日期+时间排序）
   2.3 生成 [地点][日期]建筑钢材原料价格清单.xlsx
   2.4 生成/复用厂家对照表
   2.5 存在待确认 → 阻断并展示明细待确认
   2.6 回写 G1/G3/G4（页签先清空后标红未匹配）

3. 图片/文档流程（image_doc/both）
   3.1 提取价格和库存（所有文件类型统一使用MiniMax视觉识别）
       - .jpg/.png → MiniMax视觉识别（提取价格+库存）
       - .txt      → 文本解析（提取价格+库存）
       - 补充.txt  → 解析调整规则（下30/电议等）
   3.2 生成/复用图片文档对照表
   3.3 存在待确认 → 阻断并展示明细待确认
   3.4 回写 H1/H3/H4（禁止修改页签颜色）
   3.5 库存颜色标注（充足=蓝/告警=黄/缺货=红）

4. 生成总结报告与分阶段报告

5. 核对报告并在对话中详细回报结果
```

#### 5.1.2 待确认阻断后的标准恢复流程

当执行过程中出现 `pending_confirmation` 时，必须按以下流程在对话中完成：

```
步骤1：读取待确认对照并在对话中展示
   - web：展示网价待确认明细
   - image_doc：展示图片/文档待确认明细
   - both：同时展示两套明细
   - 对照明细格式：项目sheet -> 来源厂家 -> 状态 -> 说明
   - 新厂家明细：仅列 项目sheet为空 且 状态=待确认(新厂家) 的行
   - 若某一链路缺失明细，先说明原因并补跑该链路，再进入确认

步骤2：请求用户在对话中确认
   - 可"全部确认"，也可逐条修改

步骤3：根据用户确认写回待确认文件
   - 待确认匹配 -> 已确认匹配
   - 待确认(新厂家) -> 已确认不更新（除非用户明确指定其它处理）
   - 若同一项目sheet出现重复映射行，先按优先级去重后再执行回写

步骤4：运行 check_pending_mapping.py 确认待确认为 0

步骤5：重新执行同一命令继续

⚠️ 禁止事项：执行者不得绕过对话确认，直接把待确认改成已确认
```

### 5.2 厂家对照确认机制（增量确认）

**核心原则**：
- 已确认的对照关系会**持久化保存**到 `运行产物/<项目>/厂家对照表_*_正式.json`
- 后续运行时**自动复用**已确认的对照，无需重复确认
- 仅当出现**新厂家**时才触发确认流程

**增量确认逻辑**：
```
1. 加载已有的"已确认"对照表（厂家对照表_*_正式.json / 图片文档*已确认.json）
2. 对比三方数据源：
   - 报价表Excel中的所有Sheet
   - 网价清单中的所有厂家
   - 图片/文档报价中的所有厂家
3. 处理结果：
   - ✓ 厂家已在确认表中 → 直接复用，跳过确认
   - ✗ 发现新厂家 → 生成"待确认"记录，在对话中展示标准格式明细
```

**确认触发条件**：
- 项目报价表Excel中出现新的Sheet（厂家）
- 网价清单中出现新的厂家
- 图片/文档报价中出现新的厂家

### 5.3 待确认展示规则

- 若出现厂家对照待确认：
  - `web` 必须展示网价对照明细；
  - `image_doc` 必须展示图片/文档对照明细；
  - `both` 必须同时展示两套明细。
- **完整清单展示规则**：
  - 必须展示所有厂家，不能遗漏任何厂家
  - 必须按状态分类展示，格式：
    ```
    | 序号 | 项目Sheet | 来源Sheet | 状态 | 说明 |
    |------|----------|----------|------|------|
    | 1 | xxx | xxx | 待确认匹配/未匹配/新厂家等 | 具体说明 |
    ```
  - 每类状态必须汇总数量：`待确认匹配X家 | 未匹配X家 | 新厂家X家 | 跳过X个`

### 5.4 无新增厂家告知规则

- 若本轮执行后**未发现任何新增待确认项**（即所有厂家的对照关系均已在前序运行中确认过），
  必须在对话结果中**明确告知用户**：
  > "✅ 本次未发现新厂家，所有对照关系已复用历史确认记录，无需额外操作。"
- 告知内容应包含：复用的已确认匹配数量、跳过数量等关键统计信息
- **禁止**在无新增厂家时保持沉默或省略此告知

### 5.5 更新完成结果回报规则

- 更新完成后必须详细回报：
  - 已更新明细（所有厂家，不能遗漏）
  - 未更新明细及具体原因（不可省略）
  - 价格对比格式：`3690 → 3690`（同时显示老价格和新价格）
- 格式要求：
  ```
  | 序号 | 厂家 | G3(盘螺) 老→新 | G4(螺纹) 老→新 |
  |------|------|----------------|----------------|
  | 1 | 马钢 | 3690 → 3690 | 3430 → 3430 |
  ```
- 未更新原因格式：`跳过（X个）：[厂家名1、厂家名2] - [原因]`
- 原因分类：未匹配、不更新、未提取到有效价格、来源缺失、登录失败、网络异常等

### 5.6 映射唯一性规则

- 同一 `项目文件Sheet` 若出现重复映射，回写前自动去重，优先级：
  - `已确认匹配` > `已确认不更新` > `待确认*` > `未匹配*` > 其它

### 5.7 必须遵守的关键规则

| 规则 | 说明 |
|------|------|
| **网价链路** | 先登录再查询；先清单、再对照、再回写 |
| **对照链路** | 优先复用、增量确认；新厂家必须先确认 |
| **对话确认链路** | 待确认阶段必须由用户在对话中确认，禁止执行者私自确认 |
| **对话展示链路** | 单文件/批量、`web`/`image_doc`/`both` 均必须按执行链路展示待确认厂家对照明细 |
| **结果回报链路** | 单文件/批量、`web`/`image_doc`/`both` 均必须详细列出更新结果，未更新项必须说明原因 |
| **页签颜色** | 网价"先清空后标红"；图片/文档禁止改页签颜色 |
| **细粒度地点** | 未命中逐级回退；冲突时必须向用户确认 |
| **库存颜色** | 充足=蓝色、告警=黄色、缺货=红色；只标第一列（价格列），不标网差列 |

## 6. 常见问题

- 登录失败：先检查 `网站账号密码.txt` 与站点登录态/风控。
- MiniMax API异常：检查 `.env` 中的 `MINIMAX_API_KEY` 是否配置正确。
- 若报告里有 `pending_confirmation`：先确认待确认对照，再重跑同一命令。
- 若想强制重新匹配：删除 `运行产物/*/厂家对照表_*_正式.json` 和 `运行产物/*/*已确认.json`，再执行更新。
